# 疲劳驾驶监测系统 - 技术路线说明书

## 一、 项目背景与核心目标

本项目旨在开发一套**软硬件结合的实时疲劳驾驶监测系统**。其核心目标是通过可穿戴硬件（如智能眼镜或头带）采集驾驶员的眼睑电压（或类似生理电信号），利用前端的 Web Bluetooth 技术将信号接入系统，交由后端的数字信号处理（DSP）算法进行分析，最终在前端大屏上实时呈现驾驶员的疲劳状态、眨眼特征以及危险预警，从而达到预防交通事故的目的。

---

## 二、 系统整体架构设计

系统采用**前后端分离**与**边缘/云端协同**的架构思想。整体架构分为四个逻辑层：

1. **感知层（Hardware / Sensor Layer）**
   - **硬件选型**：搭载蓝牙低功耗（BLE）模块的单片机（如 ESP32 或 nRF 系列）及贴片电极/光电传感器。
   - **核心职责**：以固定采样率（如 100Hz）采集眼睑电压模拟信号，进行 A/D 转换，并通过 BLE GATT 协议（特定 UUID 的特征值）向外广播二进制字节流。

2. **接入层（Data Acquisition Layer）**
   - **实现方案**：基于现代浏览器的 **Web Bluetooth API**。
   - **核心职责**：前端网页直接搜索并连接硬件设备，将接收到的二进制流（ArrayBuffer）解析为浮点数数组（Float Array），并将其打包装载为 JSON 格式。

3. **计算与中转层（Algorithm & Backend Layer）**
   - **实现方案**：Python + Flask + NumPy/SciPy。
   - **核心职责**：作为本地或云端的算法引擎。接收来自接入层的原始数据，将其放入时序滑动窗口，调用核心疲劳算法进行科学计算。同时作为数据总线，将计算结果（如疲劳分、眼睑状态）通过 WebSocket 广播出去。

4. **展现与交互层（Presentation Layer）**
   - **实现方案**：React 18 + Vite + Tailwind CSS + Recharts。
   - **核心职责**：维持与后端的 WebSocket 长连接，实时消费计算结果。利用高性能图表库渲染生理信号波形与疲劳趋势图，并在达到危险阈值时触发视觉/听觉警报。

## 三、 核心技术栈与选型依据

### 1. 算法与后端（Python 生态）
- **Flask & Flask-Sock**：选用 Flask 是因为其极度轻量，适合快速构建原型并暴露 HTTP/WebSocket 接口。`flask-sock` 提供了原生的 WebSocket 支持，实现低延迟的双向通信。
- **NumPy & SciPy**：在生理信号处理领域，这两个库是行业标准。`NumPy` 提供了高效的 C 语言级矩阵运算（如数组切片、统计）；`SciPy` 的 `signal` 和 `ndimage` 模块提供了成熟的低通滤波、寻峰（Peak Detection）和形态学运算，是提取眨眼特征的算力基石。

### 2. 前端展示（现代 Web 生态）
- **React 18 & Vite**：React 提供了组件化的状态管理能力，适合处理复杂的数据流刷新；Vite 提供了极速的冷启动和 HMR（热更新），极大提升开发效率。
- **Web Bluetooth API**：抛弃了传统的“串口助手”或“本地客户端”方案。让网页直接获取蓝牙权限，实现了**“零安装、开箱即用”**的极致用户体验，是 Web IoT 领域的前沿实践。
- **Tailwind CSS & Radix UI**：Tailwind 提供了原子化 CSS，保证了样式的可控性与极小体积；Radix UI 提供了无头（Headless）的可访问组件，确保了复杂的弹窗、Tabs 交互的流畅性。
- **Recharts**：基于 D3 和 SVG 构建，能非常直观、美观地渲染实时折线图。

## 四、 核心算法技术实现路径

核心算法模块（位于 `algorithm/blink_fatigue.py`）是本系统的“大脑”。其数据处理链路严格遵循经典数字信号处理（DSP）流程：

### 阶段一：信号预处理 (Preprocessing)
1. **去直流与基线校准**：去除原始电压信号的直流偏置。
2. **形态学基线漂移消除**：驾驶过程中由于头部运动、流汗等原因，基线会发生缓慢漂移。采用 `scipy.ndimage` 的**灰度开运算与闭运算（Grey Opening/Closing）**，使用秒级的大窗口提取极低频的趋势线（Trend Base），将原信号减去该趋势线，从而“拉平”波形，但保留了“长闭眼”这种有价值的低频特征。
3. **低通滤波**：应用截止频率为 6.5Hz 的 8 阶 Butterworth 低通滤波器，滤除肌电干扰、工频干扰等高频噪声。
4. **归一化**：将信号映射至 `[0, 1]` 区间，统一物理语义（值越大代表眼睛越闭合）。

### 阶段二：特征提取 (Feature Extraction)
1. **动态寻峰 (Peak Detection)**：基于预设的高度阈值和时间距离，利用 `scipy.signal.find_peaks` 提取代表“眨眼”的高值点（Peaks）。
2. **谷值界定 (Valley Detection)**：在 Peaks 附近寻找局部极小值，以此精确界定一次眨眼的“起始点”和“结束点”。
3. **计算生理特征**：根据波峰和波谷，推导出核心指标：**眨眼频率（Hz）**、**平均眨眼幅度**、**平均持续时间**、以及**眼睑闭合比例（类似 PERCLOS 概念）**。

### 阶段三：事件分类与疲劳评估 (Event Classification & Assessment)
1. **事件分类**：将提取的眨眼动作划分为“正常（Normal）”、“长眨眼（Long，通常超 0.5s）”和“凝视/不完全眨眼（Incomplete/Stare）”。
2. **多维加权评分**：不依赖单一指标。采用线性加权模型：
   - 眼闭合比例得分（权重 70%）
   - 长眨眼与不完全眨眼比例得分（各占 10%）
   - 频率与持续时间变异得分（各占 5%）
3. **状态输出**：将最终得分（0-100）映射为“清醒”、“轻度”、“中度”、“严重”四个疲劳等级，推给业务层。

## 五、 数据链路与通信机制

系统设计了两条平行的数据输入通道，确保了系统在开发、测试和生产环境下的灵活性。

1. **真实硬件流 (生产/展示模式)**
   - 驾驶员佩戴硬件 $\rightarrow$ 手机/电脑 Chrome 浏览器通过 `navigator.bluetooth` 连接 $\rightarrow$ 前端解析二进制流 $\rightarrow$ 前端通过 HTTP POST（`fetch`）高频转发至 `localhost:3001/api/bluetooth-data` $\rightarrow$ 后端写入内存 Buffer $\rightarrow$ 满足时间窗口（如 3 秒）触发算法 $\rightarrow$ 结果 JSON 通过 WebSocket `/ws` 推送回前端 $\rightarrow$ 触发 React 状态更新，图表重绘。

2. **模拟注入流 (研发/压测模式)**
   - 通过 `test_data_injection.py` 脚本 $\rightarrow$ 模拟生成带有白噪声的正常/疲劳波形 $\rightarrow$ 直接通过 HTTP POST 写入后端 `/api/bluetooth-data` $\rightarrow$ 后端处理并广播 $\rightarrow$ 前端呈现。
   - **意义**：实现了前后端的物理隔离与解耦，使得算法工程师可以在没有硬件的情况下进行极端工况的压测与参数调优。

## 六、 未来演进与优化方向

为了将项目从目前的“高可用原型”推向“工业级/商业级产品”，技术路线规划了以下演进方向：

1. **算法层：向 AI 与多模态演进**
   - **引入深度学习**：收集海量标注数据，训练 1D-CNN 或 LSTM 时序模型，利用 `ONNX Runtime` 替代当前的规则引擎，提升对非线性微小疲劳特征的捕捉能力。
   - **多模态融合**：结合摄像头图像（如嘴部打哈欠特征）或陀螺仪数据（头部点头特征）进行多维联合决策。

2. **数据层：引入时序数据库**
   - 摒弃当前的内存数组缓冲。引入 InfluxDB 等时序数据库，实现历史数据的持久化存储，支撑“驾驶员历史疲劳报告”生成以及离线算法训练。

3. **架构层：边缘计算下沉**
   - 将精简后的 C/C++ 版算法直接烧录至 ESP32 等单片机中（TinyML）。单片机直接输出疲劳分数，前端仅作为展示面板。这将彻底砍掉对 PC 后端算力的依赖，极大降低延迟和部署成本。

4. **前端体验层：PWA 改造**
   - 配置 `vite-plugin-pwa`，使前端网页具备离线缓存能力，并允许用户将其安装到手机桌面，获得媲美原生 App 的无边框沉浸式驾驶监控体验。
